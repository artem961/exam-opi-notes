# Билет 26. Требования к удобству использования и надёжности

## Ответ

### Usability — удобство использования

Описывает, насколько легко пользователю работать с системой. Не «что делает система», а «насколько комфортно это делать».

Ключевые свойства:

```
┌─────────────────────────┬──────────────────────────────────────────────┐
│ Свойство                │ Пример требования                            │
├─────────────────────────┼──────────────────────────────────────────────┤
│ Время обучения          │ Новый пользователь должен освоить основные   │
│                         │ операции за 30 минут                         │
├─────────────────────────┼──────────────────────────────────────────────┤
│ Эргономика интерфейса   │ Ключевые действия — не более 3 кликов        │
├─────────────────────────┼──────────────────────────────────────────────┤
│ Документация            │ Система должна иметь встроенную справку      │
├─────────────────────────┼──────────────────────────────────────────────┤
│ Стандарты интерфейса    │ Интерфейс должен соответствовать             │
│                         │ корпоративным UI-гайдлайнам                  │
└─────────────────────────┴──────────────────────────────────────────────┘
```

### Reliability — надёжность

Описывает способность системы работать без сбоев в течение заданного времени при заданных условиях.

Ключевые метрики и свойства:

```
┌──────────────────────┬───────────────────────────────────────────────────┐
│ Свойство             │ Пример требования                                 │
├──────────────────────┼───────────────────────────────────────────────────┤
│ MTBF                 │ Среднее время между отказами ≥ 720 часов          │
│ (Mean Time Between   │                                                   │
│  Failures)           │                                                   │
├──────────────────────┼───────────────────────────────────────────────────┤
│ Частота отказов      │ Не более 1 сбоя в месяц в штатном режиме         │
├──────────────────────┼───────────────────────────────────────────────────┤
│ Восстановление       │ После сбоя система должна восстановиться          │
│                      │ в течение 5 минут без потери данных               │
├──────────────────────┼───────────────────────────────────────────────────┤
│ Точность вычислений  │ Расчёт итоговой суммы — с точностью до 0,01 руб. │
├──────────────────────┼───────────────────────────────────────────────────┤
│ Доступность          │ Uptime ≥ 99,9% (не более 8,7 часов простоя/год)  │
└──────────────────────┴───────────────────────────────────────────────────┘
```

## Подробно

### Почему Usability — это тоже требования

Разработчики часто считают удобство интерфейса «субъективным» и не включают его в спецификацию. Это ошибка: удобство можно измерить и проверить. Юзабилити-тест — это эксперимент: N пользователей, M задач, фиксируем время выполнения и количество ошибок. Если результат не соответствует требованию — система не прошла тест.

### Что считается сбоем

Сбой — это когда система не выполняет свою функцию в допустимых условиях. Важно различать:
- **Дефект (defect)** — ошибка в коде или документации.
- **Сбой (fault)** — проявление дефекта при выполнении.
- **Отказ (failure)** — наблюдаемое пользователем неправильное поведение.

Требования к надёжности пишут через отказы, потому что именно их видит пользователь.

### MTBF на практике

MTBF = общее время работы / количество отказов.

Пример: если система за год (8760 часов) упала 12 раз, MTBF = 8760 / 12 = 730 часов. Требование «MTBF ≥ 720 часов» выполнено.

### Доступность vs надёжность

Доступность (availability) — вероятность того, что система работает в произвольный момент времени. Вычисляется как:

$$\text{Availability} = \frac{MTBF}{MTBF + MTTR}$$

где MTTR (Mean Time To Recovery) — среднее время восстановления. Чтобы повысить доступность, нужно либо уменьшать частоту сбоев (растить MTBF), либо ускорять восстановление (уменьшать MTTR).
