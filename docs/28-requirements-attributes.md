# Билет 28. Атрибуты требований

## Ответ

**Атрибуты требований** — метаданные, которые сопровождают каждое требование и позволяют управлять им на протяжении всего проекта.

### Основные атрибуты

| Атрибут | Значения / пример |
|---------|-------------------|
| **Идентификатор** | REQ-042 — уникальный код, по которому ищут требование |
| **Статус** | Предложено / Одобрено / Реализовано / Проверено / Отклонено |
| **Приоритет (MoSCoW)** | Must / Should / Could / Won't |
| **Трудоёмкость** | Оценка в story points или человеко-часах |
| **Стабильность** | Стабильное / Изменчивое — вероятность изменений |
| **Ответственный** | Кто отвечает за реализацию |
| **Источник** | Кто сформулировал: заказчик X, встреча от 10.01 |

### Приоритизация: метод MoSCoW

```
Must have   — обязательно в текущей версии; без этого система бесполезна
Should have — важно, но можно временно обойтись; реализуется если есть время
Could have  — желательно; реализуется только если всё остальное готово
Won't have  — не в этой версии, возможно — в следующей
```

**Пример:**
```
REQ-01  [Must]    Система должна аутентифицировать пользователя.
REQ-07  [Should]  Система должна помнить пользователя между сессиями.
REQ-15  [Could]   Система должна поддерживать вход через Google.
REQ-22  [Won't]   Система должна поддерживать биометрическую аутентификацию.
```

## Подробно

### Зачем нужны атрибуты

Без атрибутов требования — просто список. С атрибутами — управляемый беклог. Атрибуты позволяют:
- Принять решение, что реализовывать в первую очередь (приоритет + трудоёмкость).
- Отследить состояние каждого требования (статус).
- Понять, что изменится, если пересмотреть требование (трассировка + источник).
- Управлять рисками: изменчивые требования требуют буфера по времени.

### Статус и его переходы

```
Предложено → Одобрено → Реализовано → Проверено → Закрыто
                ↓
            Отклонено
```

Требование не считается реализованным, пока не прошло проверку (тестирование). «Реализовано» и «Проверено» — разные состояния.

### MoSCoW на практике

Типичная ошибка — всё пометить как Must. Тогда метод теряет смысл. Хорошая практика: Must — не более 60% общего объёма функциональности. Это оставляет буфер на непредвиденные задержки.

### Стабильность требования

Стабильность — прогноз того, изменится ли требование. Изменчивые требования (например, «интеграция с платёжной системой X, которую ещё обсуждают с партнёром») рискованно реализовывать рано: придётся переделывать. Атрибут стабильности помогает планировщику откладывать нестабильные требования на более поздние итерации.

### Трудоёмкость и планирование

Трудоёмкость — оценка объёма работ, необходимых для реализации требования. В Agile используют относительные единицы (story points), в классических проектах — человеко-часы. Атрибут нужен для планирования итераций: зная приоритет и трудоёмкость, менеджер формирует план спринта.
