# Билет 68. Пирамида памяти и её влияние на производительность

## Ответ

**Иерархия памяти (пирамида памяти)** — упорядоченный набор уровней хранения данных, которые различаются по скорости доступа, стоимости и объёму.

### Уровни иерархии

```
         /\
        /  \   Регистры CPU (~256 Б)  — 0.3 нс
       /────\
      / L1   \  (~32–64 КБ)           — 1 нс
     /────────\
    /   L2     \ (~256 КБ – 1 МБ)     — 4 нс
   /────────────\
  /     L3       \ (~8–64 МБ)         — 10–40 нс
 /────────────────\
/ Оперативная RAM  \ (ГБ)             — 60–100 нс
────────────────────
       SSD          (ТБ)              — 50–200 мкс
────────────────────
       HDD          (ТБ)              — 5–10 мс
```

Чем ниже уровень — тем медленнее, дешевле, больше.

### Влияние на производительность

**Cache hit** — данные найдены в L1/L2/L3 → быстро.  
**Cache miss** — данных нет в кэше → загрузка из RAM → медленнее в 100× раз относительно L1.  
**Page fault (major)** — данных нет в RAM → загрузка с диска → медленнее в 100 000× раз.

### Принципы локальности

**Временна́я локальность (temporal locality):** данные, к которым обращались недавно, скоро понадобятся снова. Кэш хранит «горячие» данные.

**Пространственная локальность (spatial locality):** данные, соседние с обращением, тоже скоро понадобятся. Кэш загружается **строками** (cache line, 64 байта): при обращении к байту N загружаются байты N–N+63.

```
Хороший паттерн (sequentially):   a[0], a[1], a[2], ...  → все в одной строке
Плохой паттерн (random access):   a[0], a[100], a[50], ... → промах на каждом шаге
```

## Подробно

### Числа, которые нужно знать

| Уровень | Время доступа | Относительно |
|---------|--------------|--------------|
| L1 cache | 1 нс | 1× |
| L2 cache | 4 нс | 4× |
| L3 cache | 40 нс | 40× |
| RAM | 100 нс | 100× |
| SSD (random read) | 150 мкс | 150 000× |
| HDD (random read) | 10 мс | 10 000 000× |
| Сеть (RTT в дата-центре) | 500 мкс | 500 000× |

### Cache-friendly код

**Плохой пример: обход матрицы по столбцам**

```c
// Cache-unfriendly: прыгаем по памяти с шагом N
for (int j = 0; j < N; j++)
    for (int i = 0; i < N; i++)
        sum += a[i][j];  // a[0][j], a[1][j] — далеко друг от друга
```

**Хороший пример: обход по строкам**

```c
// Cache-friendly: последовательный доступ
for (int i = 0; i < N; i++)
    for (int j = 0; j < N; j++)
        sum += a[i][j];  // a[i][0], a[i][1] — рядом в памяти
```

Разница в производительности на больших матрицах: 5–10×.

### False sharing (ложное разделение)

Два потока работают с разными переменными, но они оказываются в одной cache line. Каждый поток, обновляя «свою» переменную, инвалидирует строку кэша другого потока → кэш-промах без видимой причины.

Решение: выровнять структуры данных так, чтобы переменные разных потоков попали в разные cache lines (padding до 64 байт).

### Swap и TLB

**Swap:** когда RAM заполнена, ядро вытесняет «холодные» страницы на диск. Доступ к вытесненной странице (major page fault) занимает миллисекунды — в 100 000 раз медленнее RAM. Система начинает «тормозить».

**TLB (Translation Lookaside Buffer):** кэш трансляции виртуальных адресов в физические. Промах TLB → обращение к page table в памяти. Использование huge pages (2 МБ вместо 4 КБ) снижает число TLB-промахов.
