# Билет 68. Пирамида памяти и её влияние на производительность

## Ответ

**Пирамида памяти (иерархия памяти)** — упорядоченный набор уровней хранения данных. Чем выше уровень — тем быстрее, дороже и меньше объём; чем ниже — тем медленнее, дешевле и больше объём.

### Пирамида памяти (по лекции)

| Уровень | Объём | Время доступа (Тд) | Аналогия (такт ≈ 1 с диалога) | Тип | Управление |
|---------|-------|--------------------|-------------------------------|-----|------------|
| **CPU (регистры)** | 100–1000 б | < 1 нс | 1 с | Регистр | компилятор |
| **L1 Cache** | 32–128 Кб | 1–4 нс | 4 с | Ассоциативный | аппаратура |
| **L2–L3 Cache** | 0.5–32 Мб | 8–20 нс | 30 с | Ассоциативный | аппаратура |
| **Основная память** | 0.5 Гб–4 Тб | 60–200 нс | 3–10 мин | Адресная | malloc, free, slab-alloc |
| **SSD** | 128 Гб–1 Тб / drive | 25–250 мкс | 5 дней | Блочный | open, read, write |
| **Жёсткие диски** | 0.5 Тб–4 Тб / drive | 8–20 мс | 4 месяца | Блочный | open, read, write |
| **Магнитные ленты** | 1–6 Тб / картридж | 1–260 с | 200 лет | Последовательный | программно |

> **NB:** при 2% cache miss производительность падает ниже 60% от идеальной.

Обращение к основной памяти стоит **сотни циклов** процессора, к диску — **миллионы циклов**. Поэтому минимизация обращений к нижним уровням пирамиды повышает производительность в сотни и даже миллионы раз.

### Аналогия с диалогом

Если отождествить такт процессора со скоростью межличностного общения (1 такт ≈ 1 секунда), то на вопрос «Как дела?»:

- **регистр** ответит через 1 с — нормальная беседа;
- **L1** — через 4 с — ещё приемлемо;
- **L2** — через 30 с — уже странно;
- **основная память** — через минуты (собеседник «задумался о чём-то другом»);
- **SSD и жёсткие диски** — дни и месяцы (неприемлемо: «он что, соврать собрался?»);
- **магнитные ленты** — сотни лет.

### Влияние на производительность

**Cache hit** — данные найдены в L1/L2/L3 → быстро.
**Cache miss** — данных нет в кэше → загрузка из RAM → медленнее в сотни раз относительно L1.
**Page fault (major)** — данных нет в RAM → загрузка с диска → медленнее в миллионы раз.

Учёт архитектуры ЭВМ позволяет сделать программы быстрее **на несколько порядков**.

### Принципы локальности

**Временна́я локальность (temporal locality):** данные, к которым обращались недавно, скоро понадобятся снова. Кэш хранит «горячие» данные.

**Пространственная локальность (spatial locality):** данные, соседние с обращением, тоже скоро понадобятся. Кэш загружается **строками** (cache line, 64 байта): при обращении к байту N загружаются байты N…N+63.

```
Хороший паттерн (последовательно): a[0], a[1], a[2], ...  → все в одной строке
Плохой паттерн (random access):    a[0], a[100], a[50], ... → промах на каждом шаге
```

## Подробно

### Почему пирамида именно такая

Быстрая память дорогая и физически не может быть большой (регистры — рядом с АЛУ, кэш — на кристалле процессора). Дешёвая память большая, но далеко и медленно. Поэтому строят иерархию: маленький быстрый уровень кэширует данные большого медленного. Управление верхними уровнями (регистры, кэш) прозрачно для программиста — этим занимаются компилятор и аппаратура; нижними уровнями (память, диск, лента) управляет программа явно.

### Cache-friendly код

**Плохой пример: обход матрицы по столбцам**

```c
// Cache-unfriendly: прыгаем по памяти с шагом N
for (int j = 0; j < N; j++)
    for (int i = 0; i < N; i++)
        sum += a[i][j];  // a[0][j], a[1][j] — далеко друг от друга
```

**Хороший пример: обход по строкам**

```c
// Cache-friendly: последовательный доступ
for (int i = 0; i < N; i++)
    for (int j = 0; j < N; j++)
        sum += a[i][j];  // a[i][0], a[i][1] — рядом в памяти
```

Разница в производительности на больших матрицах: 5–10×.

### False sharing (ложное разделение)

Два потока работают с разными переменными, но они оказываются в одной cache line. Каждый поток, обновляя «свою» переменную, инвалидирует строку кэша другого потока → кэш-промах без видимой причины.

Решение: выровнять структуры данных так, чтобы переменные разных потоков попали в разные cache lines (padding до 64 байт).

### Swap и TLB

**Swap:** когда RAM заполнена, ядро вытесняет «холодные» страницы на диск. Доступ к вытесненной странице (major page fault) занимает миллисекунды — в миллионы раз медленнее RAM. Система начинает «тормозить».

**TLB (Translation Lookaside Buffer):** кэш трансляции виртуальных адресов в физические. Промах TLB → обращение к page table в памяти. Использование huge pages (2 МБ вместо 4 КБ) снижает число TLB-промахов.
