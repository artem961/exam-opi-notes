# Билет 56. Источники данных для тестирования. Роли и деятельности в тестировании

## Ответ

### Источники данных для тестирования

Откуда берут идеи для тест-кейсов:

| Источник | Что даёт |
|----------|----------|
| **Спецификация / требования (чёрный ящик)** | Тесты на соответствие функциям; не знаем как реализовано — проверяем что должно быть |
| **Исходный код (белый ящик)** | Тесты на покрытие ветвей и путей; знаем внутреннюю структуру |
| **Опыт и интуиция (серый ящик)** | Эвристические тесты; «это место всегда ломается» |
| **UML-модели** | Прецеденты → тест-сценарии; диаграммы состояний → переходы между состояниями |
| **Дефекты из прошлых проектов** | Регрессионные тесты на ранее исправленные баги |

**Чёрный ящик** — тестировщик не видит код, тестирует интерфейс.  
**Белый ящик** — тестировщик видит код, покрывает внутреннюю логику.

### Роли в тестировании

| Роль | Обязанности |
|------|-------------|
| **Test Manager** | Планирует тестирование, распределяет ресурсы, отчитывается о качестве |
| **Test Designer** | Разрабатывает тест-кейсы и тест-стратегию |
| **Tester** | Выполняет тесты, фиксирует дефекты |
| **Test Automation Engineer** | Пишет автоматизированные тесты и поддерживает тест-фреймворк |
| **Developer** | Пишет юнит-тесты и устраняет найденные дефекты |

### Деятельности в тестировании

```
Планирование → Проектирование тестов → Подготовка окружения
→ Выполнение тестов → Оценка результатов → Отчёт о качестве
```

## Подробно

### Почему важны несколько источников

Разные источники дополняют друг друга. Тесты только по требованиям пропустят ошибки реализации (нужен белый ящик). Тесты только по коду могут упустить, что реализована не та функция (нужен чёрный ящик). Опыт тестировщика находит то, что не описано нигде — граничные случаи, нетипичные комбинации.

### UML-модели как источник тестов

- **Use case diagram** → набор сценариев использования → основной поток + альтернативные потоки.
- **State diagram** → переходы между состояниями → тест на каждый переход.
- **Sequence diagram** → порядок вызовов → тесты на правильный порядок.

Пример: диаграмма состояний аутентификации (idle → authenticating → authenticated → error) даёт как минимум 4 теста: успешный вход, неверный пароль, повтор после ошибки, выход.

### Тест-план

Документ, описывающий стратегию тестирования:
- Что тестируется (scope).
- Что не тестируется (out of scope).
- Какие уровни тестирования.
- Критерий начала и окончания тестирования.
- Риски тестирования.

### Метрики тестирования

- Количество найденных дефектов по этапам.
- Плотность дефектов (дефектов / 1000 строк кода).
- Покрытие требований тестами (% требований с тестами).
- Процент прохождения тестов.
