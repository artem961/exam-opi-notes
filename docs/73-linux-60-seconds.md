# Билет 73. Системный анализ Linux «за 60 секунд»

## Ответ

Методика Брендана Грегга — 10 команд, которые нужно выполнить при получении доступа к медленному серверу. Выполняются последовательно, каждая занимает несколько секунд.

### Чеклист «60 секунд»

```bash
# 1. Нагрузка системы: load average и uptime
uptime

# 2. Сообщения ядра: ошибки, OOM, проблемы с оборудованием
dmesg -T | tail -20

# 3. Память и swap в динамике
vmstat 1 5

# 4. Загрузка CPU по ядрам
mpstat -P ALL 1 3

# 5. CPU/память по процессам
pidstat 1 5

# 6. Статистика дисков
iostat -xz 1 3

# 7. Свободная RAM и swap
free -m

# 8. Сетевая статистика и ошибки
sar -n DEV 1 3

# 9. Список активных TCP-соединений
ss -s

# 10. Топ процессов по CPU/памяти
top
```

### Что ищем на каждом шаге

| Шаг | Команда | Тревожный признак |
|-----|---------|-------------------|
| 1 | `uptime` | load >> числа ядер |
| 2 | `dmesg` | OOM kills, I/O errors, ext4 errors |
| 3 | `vmstat` | si/so > 0 (swap), b > 0 (заблокированы I/O) |
| 4 | `mpstat` | iowait > 20%, steal > 5% (виртуализация) |
| 5 | `pidstat` | один процесс = весь %cpu |
| 6 | `iostat` | %util = 100%, await > 10мс |
| 7 | `free` | available < 100МБ, swap used > 0 |
| 8 | `sar -n` | высокий %ifutil, ошибки пакетов |
| 9 | `ss -s` | десятки тысяч соединений |
| 10 | `top` | подтвердить подозрения |

### Интерпретация результатов

```
load высокий + %iowait высокий  → диск или сеть
load высокий + %user высокий    → приложение грузит CPU
load высокий + %idle высокий    → блокировки (lock contention)
swap используется               → нехватка RAM
dmesg: "oom-kill"               → процессы убиваются из-за нехватки RAM
```

## Подробно

### mpstat vs top

`top` показывает сумму по всем ядрам. `mpstat -P ALL` — по каждому ядру отдельно. Если одно ядро 100%, а остальные 5% → однопоточное приложение = бутылочное горло.

```bash
mpstat -P ALL 1 1
CPU    %usr  %sys  %iowait  %idle
all    25.0   5.0      2.0   68.0
0      95.0   3.0      1.0    1.0  ← ядро 0 перегружено
1       5.0   2.0      1.0   92.0
2       4.0   2.0      1.0   93.0
3       3.0   2.0      1.0   94.0
```

### %steal — виртуализация

`%steal` — CPU, который гипервизор отдаёт другим виртуальным машинам. Если steal > 5% — ВМ «перегружена» на физическом хосте, соседние ВМ забирают ресурсы.

### Полная методология USE

Метод USE (Brendan Gregg) — для каждого ресурса проверить три метрики:
- **U**tilization (использование)
- **S**aturation (насыщение/очередь)
- **E**rrors (ошибки)

Для CPU: utilization = %user+%sys, saturation = load average / cores, errors = machine check exceptions.

Для диска: utilization = %util, saturation = avgqu-sz (длина очереди), errors = `iostat` errors count.

### После 60 секунд

Если 60-секундный анализ не дал ответа, следующий шаг зависит от найденного:

- Высокий %user → профилировщик приложения (perf, async-profiler).
- Высокий %iowait → `iotop`, `blktrace` — трассировка I/O до конкретных файлов.
- Много соединений → `ss -tp` — кто держит соединения, `tcpdump` — что по ним летит.
- Рост памяти → `smem` — реальное потребление памяти по процессам.
