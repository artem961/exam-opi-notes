# Билет 76. Компромиссы (trade-offs) в производительности

## Ответ

В производительности нет бесплатных решений: каждое улучшение одного параметра ухудшает другой. Важно осознанно выбирать компромисс под конкретные требования.

### Основные компромиссы

#### CPU vs Память

| Решение | Что улучшает | Что ухудшает |
|---------|-------------|--------------|
| Кэширование результатов | CPU (не пересчитывать) | Память (хранить кэш) |
| Предвычисление (lookup table) | CPU (заменить вычисление чтением) | Память (таблица) |
| Сжатие данных | Память, I/O | CPU (декомпрессия) |
| JIT-компиляция | CPU (быстрый код) | Память (скомпилированный код) |

#### Latency vs Throughput

| Подход | Latency | Throughput |
|--------|---------|-----------|
| Пакетная обработка (batching) | Хуже (ждём батч) | Лучше (меньше overhead/запрос) |
| Конвейеризация (pipelining) | Хуже (первый элемент долго) | Лучше (все стадии заняты) |
| Асинхронность | Лучше (не блокируем) | Лучше (больше параллелизма) |
| Синхронный вызов | Проще разработка | Хуже (блокировка) |

#### Производительность vs Надёжность

| Оптимизация | Что теряем |
|------------|-----------|
| Отказаться от `fsync()` | Скорость записи ↑, но потеря данных при сбое |
| Eventual consistency | Производительность ↑, но данные временно несогласованны |
| Кэш без инвалидации | Скорость ↑, но устаревшие данные |
| Без транзакций | Скорость ↑, но нет атомарности |

#### Производительность vs Сопровождаемость

| Оптимизация | Что теряем |
|------------|-----------|
| Инлайн-код вместо функций | Скорость ↑, но повторение кода |
| Битовые операции вместо логических | Скорость ↑, но нечитаемость |
| Ручное управление памятью | Скорость ↑, но сложность и ошибки |

### Принцип

Оптимизация производительности — **не бесплатна**. Всегда явно указывай, что жертвуешь ради скорости. Оптимизировать только там, где измерения доказали узкое место.

## Подробно

### Кэш — главный компромисс CPU vs Память

**Без кэша:**
```
Запрос → [сложное вычисление, 200 мс] → Ответ
```
**С кэшем (Redis/Memcached):**
```
Запрос → [проверить кэш: 1 мс] → Ответ (cache hit, ~99%)
          ↓ cache miss
     [вычисление 200 мс] → [сохранить в кэш] → Ответ
```

Выигрыш: 200× по latency. Расход: дополнительная RAM + сложность инвалидации.

**Проблема инвалидации:** «В Computer Science есть две по-настоящему сложные вещи: инвалидация кэша и придумывание имён» (Phil Karlton).

### Батчинг — классический throughput vs latency

Вместо 1000 INSERT по одному — INSERT с 1000 строками:

```sql
-- Медленно (1000 транзакций):
INSERT INTO t VALUES (1); INSERT INTO t VALUES (2); ...

-- Быстро (1 транзакция):
INSERT INTO t VALUES (1), (2), ..., (1000);
```

Throughput вырастает в 10–100×. Latency первой строки — выше (ждём набора батча).

### Пространство vs Время (Space-Time Trade-off)

**Мемоизация:**
```python
cache = {}
def fib(n):
    if n in cache:
        return cache[n]
    result = fib(n-1) + fib(n-2)
    cache[n] = result
    return result
```
Время: O(n) вместо O(2ⁿ). Память: O(n) для кэша.

**Хэш-таблица vs Бинарный поиск:**
- Хэш-таблица: O(1) поиск, но память на таблицу + накладные расходы на хэш.
- Отсортированный массив + бинарный поиск: O(log n) поиск, но O(1) память на элемент.

### Нормализация БД vs Производительность

**Нормализованная схема** (3NF): минимум дублирования, но JOIN при каждом запросе.  
**Денормализованная схема** (c дублированием): быстрее чтение, но дороже UPDATE и больше место.

OLTP-системы (транзакционные) → нормализация.  
OLAP-системы (аналитика) → денормализация (star schema, column store).

### Compression Trade-off

Алгоритмы сжатия:
- **Быстрое (LZ4, Snappy):** скорость сжатия/декомпрессии 400–1000 МБ/с, степень сжатия 2–3×.
- **Высокое сжатие (gzip, zstd):** степень сжатия 5–10×, но медленнее.

Если диск — узкое место → лучше сжатие (меньше I/O).  
Если CPU — узкое место → лучше без сжатия.
