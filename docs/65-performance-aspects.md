# Билет 65. Аспекты быстродействия системы. Влияние средств измерения на результаты

## Ответ

### От чего зависит скорость выполнения программ

Скорость работы системы зависит от четырёх аспектов:

| Аспект | Что включает |
|--------|--------------|
| **Системный и архитектурный** | Архитектура (монолитная, n-tier, потоковая), виртуализация и кластеризация. Требуется баланс производительности всех уровней; в трёхзвенной архитектуре медленный слой данных тормозит бизнес-логику. |
| **Низкоуровневый аппаратный** | Тактовая частота, число процессоров и ядер; объём и скорость кэшей, памяти, дисковой подсистемы. |
| **Программный** | Используемые алгоритмы и структуры данных (пузырьковая сортировка vs быстрая сортировка), пулы, многопоточность и блокировки. |
| **Человеческий фактор** | Ошибки: неправильно принятые архитектурные решения, узкое знание предметной области. |

Системный и архитектурный аспект важен в первую очередь: нужно учитывать, что в вычислительной среде уже работают другие программы, а сама среда построена по определённой архитектуре, задающей характеристики выполняемых в ней программ.

### Влияние средств измерения на результаты

Любой анализ производительности — это **научный эксперимент**, который выполняется по шагам:

1. Выбрать **критерии оценки** — параметры, по которым оценивается производительность.
2. Выбрать **средства измерения**.
3. Выбрать **нагрузку** и/или нагрузить систему.
4. Провести **анализ результатов**.
5. Изменить **один параметр** / переписать **один участок** кода.
6. **Повторять** до полного удовлетворения результатом.

Средства измерения по степени влияния на результат делятся на три класса:

| Класс | Влияние на результат | Аналогия |
|-------|----------------------|----------|
| **Невтрузивные** (non-intrusive) | Не влияют на результаты | — |
| **Слабоинтрузивные** | Вносят небольшие искажения | вольтметр в измеряемом напряжении |
| **Интрузивные** | Существенно искажают результат | весы у нечестного продавца «на руку» |

**Важно:** применительно к вычислительным системам практически любое измерение так или иначе влияет на результат. Например, счётчики производительности ОС работают «незаметно» (они всё равно собираются на системном уровне), но если бы их не собирали, это процессорное время отдавалось бы пользовательской программе.

После анализа из результатов **исключают данные, наведённые средствами измерения**: например, если журнал измерений растёт со скоростью 1 МБ/с, эту нагрузку нужно исключить из производительности дисков.

## Подробно

### Latency vs Throughput — нельзя оптимизировать оба одновременно

Пакетная обработка повышает throughput, но увеличивает latency. Запрос ждёт накопления батча.

Стриминговая обработка даёт низкую latency, но меньший throughput (накладные расходы на каждый элемент).

Выбор зависит от требований: для торговых систем важна минимальная latency, для ETL-пайплайнов — максимальный throughput.

### Sampling profiler vs Instrumentation profiler

**Sampling profiler** (например, `perf`, `async-profiler`):
- Периодически (например, 99 Гц) останавливает процесс и смотрит стек.
- Overhead: 1–5% — подходит для продакшна.
- Точность: статистическая, пропускает короткие функции.

**Instrumentation profiler** (например, JProfiler в режиме instrumentation):
- Вставляет измерения в каждый вызов функции.
- Overhead: 50–500% — только для разработки.
- Точность: полная.

### Warmup-эффект

JIT-компилятор (Java, C#, JavaScript V8) оптимизирует «горячий» код на лету. Первые секунды работы — код интерпретируется медленно, потом JIT компилирует — и становится быстрее. Измерения нужно делать после «прогрева» (warmup), иначе результаты занижены.

### Влияние GC (Garbage Collector)

В языках с GC (Java, C#, Go) GC-паузы могут резко ухудшить P99 latency. GC-цикл останавливает поток на миллисекунды — в нагрузочном тесте это даёт «хвост» медленных запросов. Профилировщик памяти помогает найти объекты, которые создаются слишком часто.
