# Билет 65. Аспекты быстродействия системы. Влияние средств измерения на результаты

## Ответ

### Аспекты быстродействия системы

**Производительность (performance)** — способность системы выполнять работу за приемлемое время при допустимом использовании ресурсов.

Основные аспекты:

| Аспект | Что характеризует | Пример требования |
|--------|-------------------|-------------------|
| **Время отклика** (Latency) | Время от запроса до ответа | «Страница загружается за < 2 с» |
| **Пропускная способность** (Throughput) | Число операций в единицу времени | «1000 транзакций/сек» |
| **Использование ресурсов** | CPU, RAM, диск, сеть | «CPU < 80% под нагрузкой» |
| **Масштабируемость** (Scalability) | Как растёт производительность при добавлении ресурсов | «Добавление сервера удваивает throughput» |
| **Доступность** (Availability) | Доля времени, когда система работает | «99.9% uptime» |

### Влияние средств измерения на результаты

**Принцип Гейзенберга в системном ПО:** средство измерения само потребляет ресурсы и искажает измерение.

```
Без профилировщика:     CPU = 45%
С профилировщиком:      CPU = 68%  ← профилировщик добавил 23%
```

Виды искажений:

| Источник искажения | Эффект |
|-------------------|--------|
| **Инструментирование кода** (instrumentation profiler) | Вставляет вызовы в каждую функцию — замедляет программу в 2–5× |
| **Логирование** | Запись в файл — дополнительные I/O операции |
| **Запуск мониторинга** (`top`, `strace`, `tcpdump`) | Потребляет CPU и кэш |
| **Heap dump** (дамп памяти) | Останавливает JVM на секунды (Stop-the-World) |
| **Трассировка системных вызовов** (`strace`) | Замедляет процесс в 10–100× |

**Вывод:** результаты измерений под нагрузкой с включёнными средствами мониторинга не отражают реального поведения системы без мониторинга.

### Правило измерения

Измерять **одну переменную** за раз в **контролируемых условиях**. Фиксировать базовую линию до включения инструмента и после, чтобы оценить overhead самого инструмента.

## Подробно

### Latency vs Throughput — нельзя оптимизировать оба одновременно

Пакетная обработка повышает throughput, но увеличивает latency. Запрос ждёт накопления батча.

Стриминговая обработка даёт низкую latency, но меньший throughput (накладные расходы на каждый элемент).

Выбор зависит от требований: для торговых систем важна минимальная latency, для ETL-пайплайнов — максимальный throughput.

### Sampling profiler vs Instrumentation profiler

**Sampling profiler** (например, `perf`, `async-profiler`):
- Периодически (например, 99 Гц) останавливает процесс и смотрит стек.
- Overhead: 1–5% — подходит для продакшна.
- Точность: статистическая, пропускает короткие функции.

**Instrumentation profiler** (например, JProfiler в режиме instrumentation):
- Вставляет измерения в каждый вызов функции.
- Overhead: 50–500% — только для разработки.
- Точность: полная.

### Warmup-эффект

JIT-компилятор (Java, C#, JavaScript V8) оптимизирует «горячий» код на лету. Первые секунды работы — код интерпретируется медленно, потом JIT компилирует — и становится быстрее. Измерения нужно делать после «прогрева» (warmup), иначе результаты занижены.

### Влияние GC (Garbage Collector)

В языках с GC (Java, C#, Go) GC-паузы могут резко ухудшить P99 latency. GC-цикл останавливает поток на миллисекунды — в нагрузочном тесте это даёт «хвост» медленных запросов. Профилировщик памяти помогает найти объекты, которые создаются слишком часто.
