# Билет 57. Понятие тестового случая и сценария

## Ответ

### Тестовый случай (Test Case)

**Тестовый случай** — минимальная единица тестирования: набор входных данных, предусловий, шагов и ожидаемых результатов для проверки одного конкретного аспекта поведения системы.

```
┌────────────────────────────────────────────────┐
│ ID:          TC-001                            │
│ Название:    Вход с верными данными            │
│ Предусловие: Пользователь зарегистрирован      │
│ Входные данные:                                │
│   Логин: user@test.com                         │
│   Пароль: correctPassword                      │
│ Шаги:                                          │
│   1. Открыть страницу входа                    │
│   2. Ввести логин и пароль                     │
│   3. Нажать «Войти»                            │
│ Ожидаемый результат:                           │
│   Пользователь перенаправлен на главную        │
│   Отображается имя пользователя                │
│ Постусловие: Сессия создана                    │
└────────────────────────────────────────────────┘
```

Структура тест-кейса: **Вход → Обработка → Выход**

### Тестовый сценарий (Test Scenario / Test Suite)

**Тестовый сценарий** — последовательность тест-кейсов, проверяющая связанный бизнес-процесс от начала до конца.

Пример: сценарий «Снятие денег в банкомате»:
1. TC-001: Вставить карту → банкомат запрашивает PIN
2. TC-002: Ввести верный PIN → доступно меню
3. TC-003: Выбрать «Снятие» → запрос суммы
4. TC-004: Ввести допустимую сумму → выдача денег, обновление баланса
5. TC-005: Ввести сумму больше баланса → отказ с сообщением об ошибке

### Разница

| | Тест-кейс | Тест-сценарий |
|-|-----------|---------------|
| Единица | Один шаг/проверка | Цепочка кейсов |
| Контекст | Изолированный | Связный процесс |
| Зависимость | Обычно независим | Кейсы зависят друг от друга |

## Подробно

### Компоненты тест-кейса подробно

**Предусловие** — состояние системы, которое должно быть выполнено до начала теста. Без выполненного предусловия тест неверен. Пример: «база данных содержит пользователя с логином user@test.com».

**Входные данные** — конкретные значения, подаваемые в систему. Должны быть точными, не «какой-то логин».

**Ожидаемый результат** — точное описание того, что должно произойти. Не «система должна работать корректно», а «HTTP-ответ 200, поле `user.name` = "Иван Иванов"».

**Постусловие** — состояние системы после выполнения теста. Важно для тестов, которые модифицируют данные: что нужно очистить?

### Принцип независимости тест-кейсов

В идеале тест-кейсы независимы: порядок их выполнения не важен, каждый можно запустить отдельно. На практике в тест-сценарии это нарушается — ступень 4 зависит от ступени 3.

### Негативные тест-кейсы

Наряду с позитивными («при верных данных — успех») обязательно пишут **негативные** («при неверных данных — правильная ошибка»):
- TC: Ввести PIN с ошибкой → «Неверный PIN, осталось 2 попытки»
- TC: Трижды ошибиться → «Карта заблокирована»

Негативные тест-кейсы часто находят больше дефектов, чем позитивные — обработка ошибок пишется в последнюю очередь и тестируется реже.
