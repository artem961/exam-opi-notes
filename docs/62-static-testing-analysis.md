# Билет 62. Техники статического тестирования. Статический анализ кода

## Ответ

### Техники статического тестирования

**Статическое тестирование** — проверка артефактов без их выполнения (см. [билет 54](54-static-dynamic-testing.md)). Три основные техники:

#### 1. Ревью (Review)
Неформальная проверка: автор показывает код или документ коллеге, коллега ищет проблемы. Без строгого процесса, без протокола. Самый быстрый и дешёвый вид.

#### 2. Walkthrough (Прохождение)
Автор **ведёт команду** по артефакту шаг за шагом. Участники задают вопросы и предлагают решения. Фокус — на понимании, а не на формальном поиске дефектов.

#### 3. Инспекция (Inspection, Fagan Inspection)
Формальный процесс с назначенными ролями, чеклистом и протоколом:

| Роль | Функция |
|------|---------|
| Автор | Отвечает на вопросы |
| Модератор | Ведёт процесс |
| Рецензент | Ищет дефекты по чеклисту |
| Регистратор | Фиксирует дефекты |

Этапы: Планирование → Обзор → Подготовка → Сессия → Переработка → Проверка.

### Статический анализ кода (инструментальный)

**Статический анализ** — автоматическая проверка кода инструментами без его запуска.

#### Виды проверок

| Вид | Что ищет | Пример инструмента |
|-----|----------|-------------------|
| **Стиль** | Нарушения соглашений | Checkstyle, ESLint |
| **Потенциальные баги** | NPE, неиспользуемые переменные | PMD, SpotBugs |
| **Уязвимости** | SQL-инъекции, XSS | FindSecBugs, Semgrep |
| **Дублирование кода** | Copy-paste | CPD (Copy-Paste Detector) |
| **Метрики** | Цикломатическая сложность | SonarQube |
| **Типы** | Ошибки типизации | mypy (Python), TypeScript |

#### Пример: SpotBugs находит NPE

```java
// Потенциальный NullPointerException:
String s = getValue();  // может вернуть null
s.length();             // NPE если s == null
// SpotBugs помечает как NP_NULL_ON_SOME_PATH
```

## Подробно

### Чеклист ревью кода

Типичные пункты:
- Имена переменных/методов понятны.
- Нет магических чисел (используются константы).
- Обработаны граничные случаи и null.
- Нет дублирования кода.
- Тесты написаны.
- Нет уязвимостей (пользовательский ввод не попадает напрямую в SQL/HTML).

### Цикломатическая сложность

Метрика, показывающая количество линейно независимых путей через метод. Считается по числу решений (`if`, `for`, `while`, `&&`, `||`).

```
Сложность = количество решений + 1
```

Значение:
- 1–5: простой, легко тестируется
- 6–10: умеренный
- 11+: сложный, требует рефакторинга

SonarQube по умолчанию предупреждает при сложности > 10 в методе.

### Когда инспекция эффективнее автоматического анализа

Автоматический анализ не понимает бизнес-логику. Инспекция кода найдёт:
- «Здесь не учитывается кейс, когда список пустой».
- «Этот алгоритм O(n²), для 10 000 элементов будет тормозить».
- «Это условие никогда не будет истинным».

### Интеграция в CI

```yaml
# GitLab CI: запускать статический анализ на каждый MR
sonar-analysis:
  stage: test
  script:
    - mvn sonar:sonar -Dsonar.host.url=$SONAR_URL
  only:
    - merge_requests
```

Блокировать слияние, если найдены критические дефекты.
