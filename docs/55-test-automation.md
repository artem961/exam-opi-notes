# Билет 55. Автоматизация тестов и ручное тестирование

## Ответ

### Ручное тестирование

**Ручное тестирование** — тестировщик сам выполняет шаги теста и проверяет результат вручную.

Применяется для: исследовательского тестирования, юзабилити-оценки, одноразовых проверок, неустойчивых функций в разработке.

### Автоматизированное тестирование

**Автоматизированное тестирование** — тесты выполняются скриптом или тест-фреймворком без участия человека.

```
Тест-скрипт → [Приложение] → Результат → Автоматическое сравнение → PASS/FAIL
```

Инструменты: JUnit, TestNG, Selenium, JMeter, Postman (Newman).

### Когда автоматизировать, когда нет

| Критерий | Автоматизировать | Вручную |
|----------|-----------------|---------|
| Частота запуска | Часто (при каждом коммите) | Редко |
| Стабильность функции | Стабильная | Меняется быстро |
| Объём | Большой (сотни тест-кейсов) | Малый |
| Проверка UX/визуала | Нет | Да |
| Исследовательское тестирование | Нет | Да |
| Регрессионное тестирование | Да | Нет |

### Пирамида тестирования

```
         /\
        /UI\         ← Мало, дорого, медленно (E2E, Selenium)
       /────\
      /  API \       ← Среднее количество (интеграционные)
     /────────\
    / Unit     \     ← Много, дёшево, быстро (JUnit)
   /────────────\
```

Большинство тестов — юнит (быстрые, дешёвые, изолированные). Меньше — интеграционных. Ещё меньше — E2E.

## Подробно

### Экономика автоматизации

Написание автоматизированного теста стоит больше, чем ручная проверка один раз. Автоматизация окупается, если тест запускается **многократно** (при каждом изменении кода).

Правило: если тест будет запущен ≥ 10 раз, автоматизация оправдана.

### Регрессионное тестирование — главный аргумент за автоматизацию

После каждого изменения нужно убеждаться, что старые функции не сломались. Вручную перепроверять всё приложение при каждом коммите невозможно. Автоматические регрессионные тесты запускаются в CI и покрывают это за минуты.

### Ограничения автоматизации

- Сложно автоматизировать проверку «выглядит ли это красиво».
- Автоматизация не заменяет исследовательское тестирование (когда тестировщик пробует неожиданные пути).
- Тесты на нестабильные функции нужно постоянно переписывать — это дорого.
- Ложные срабатывания (flaky tests) снижают доверие к набору тестов.

### Стабильность тестов

**Flaky test** — тест, который иногда проходит, иногда падает без изменений в коде. Причины: зависимость от времени, сетевые обращения, порядок тестов, глобальное состояние. Flaky тесты опасны: команда начинает игнорировать падения.
