# Билет 54. Статическое и динамическое тестирование

## Ответ

### Статическое тестирование

**Статическое тестирование** — проверка программных артефактов **без выполнения кода**.

Объекты проверки: исходный код, документация, требования, архитектурные схемы, тест-планы.

| Техника | Описание |
|---------|----------|
| **Ревью (Review)** | Неформальная проверка документа или кода коллегами |
| **Инспекция (Inspection)** | Формальная проверка по чеклисту с назначенными ролями |
| **Walkthrough** | Автор ведёт команду по артефакту, собирая вопросы |
| **Статический анализ** | Автоматический анализ кода инструментами (без запуска) |

### Динамическое тестирование

**Динамическое тестирование** — проверка программы путём **её выполнения** с конкретными входными данными.

Объекты: работающее приложение, модули, API.

```
Входные данные → [Программа] → Фактический результат
                                       ↕ сравнение
                              Ожидаемый результат
```

Дефект найден, если фактический результат ≠ ожидаемому.

### Сравнение

| Критерий | Статическое | Динамическое |
|----------|-------------|--------------|
| Выполнение кода | Нет | Да |
| Когда применяется | Любой этап (даже на уровне требований) | Только когда есть исполняемый код |
| Что находит | Логические ошибки, нарушения стандартов, противоречия в требованиях | Сбои при выполнении, неверные результаты |
| Стоимость | Низкая (нет стенда) | Выше (нужна среда) |

## Подробно

### Почему статическое тестирование ценно

Статическое тестирование позволяет найти дефекты **раньше**, чем код вообще запущен. Инспекция требований может выявить противоречие на этапе проектирования — когда исправление стоит в 10 раз дешевле, чем после кодирования.

Пример: требование «система должна поддерживать 1000 пользователей» противоречит требованию «сервер должен использовать не более 256 МБ RAM». Это выясняется при ревью требований, а не при нагрузочном тестировании.

### Виды статического анализа кода

- **Lint** — проверка стиля и очевидных ошибок (нераспределённые переменные, неиспользуемый импорт).
- **Семантический анализ** — поиск null pointer dereference, утечек памяти, buffer overflow.
- **Анализ зависимостей** — поиск циклических зависимостей и нарушений архитектуры.
- **Анализ покрытия** (без запуска) — поиск мёртвого кода.

Инструменты: SonarQube, Checkstyle, PMD (Java); ESLint (JS); Pylint (Python); Coverity, Klocwork (C/C++).

### Ролевая модель инспекции (Fagan Inspection)

| Роль | Обязанность |
|------|-------------|
| Автор | Создал артефакт; отвечает на вопросы |
| Модератор | Ведёт сессию инспекции |
| Рецензент | Читает и фиксирует дефекты |
| Регистратор | Записывает найденные дефекты |

### Статическое + динамическое = полная картина

Статический анализ найдёт неинициализированные переменные, динамическое — ошибки в бизнес-логике при реальных данных. Профессиональный процесс тестирования включает оба подхода.
