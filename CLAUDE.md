# CLAUDE.md — правила работы агента в этом репозитории

Это репозиторий конспекта к экзамену. Содержимое — Markdown-файлы в `docs/`.
Из них собирается статический сайт (MkDocs Material) и публикуется на GitHub Pages.
Твоя зона ответственности — **писать и править конспект в `docs/` и пушить в
`main`**. Сборку и публикацию делает CI и кластер автоматически.

---

## Как устроен репозиторий

```
docs/                 ← КОНСПЕКТ. Только сюда пишешь содержание.
  index.md            ← главная страница
  NN-slug.md          ← билеты (01-…, 02-…), нумерация = порядок в сайдбаре
  assets/             ← картинки и вложения
mkdocs.yml            ← конфиг сайта (тема, поиск, расширения). Менять осторожно.
requirements.txt      ← зависимости сборки
.github/workflows/    ← CI: на push в main собирает сайт и кладёт в ветку `site`
deploy/PAGES.md       ← заметки по деплою на Pages
scripts/new-bilet.sh  ← хелпер для создания нового билета из шаблона
```

---

## Рабочий цикл (каждое изменение конспекта)

1. Создай/правь файлы в `docs/`.
2. Проверь сборку локально — **обязательно перед коммитом**:
   ```bash
   mkdocs build --strict
   ```
   `--strict` валит сборку на битых ссылках и проблемах с навигацией.
   Если падает — чини, не коммить сломанное.
3. (опционально) живой предпросмотр: `mkdocs serve` → http://127.0.0.1:8000
4. Коммить и пушь в `main`:
   ```bash
   git add docs/
   git commit -m "билет NN: <что сделал>"
   git push
   ```
5. Дальше автоматически: CI собирает → пушит статику в ветку `gh-pages` →
   GitHub Pages публикует → сайт обновляется за ~1–2 минуты.
   **Твоя работа заканчивается на `git push`.**

## Соглашения по конспектам

- **Имена файлов**: `NN-slug.md`, где `NN` — номер билета с ведущим нулём
  (`01-`, `02-`, … `80-`). Номер определяет порядок в сайдбаре. `slug` —
  короткий латиницей через дефис.
- **Язык** — русский. Поиск настроен на русскую морфологию (`lang: ru`).
- **Заголовок файла** — один H1 (`# Билет NN. Название`). Внутри — H2/H3.
- **Определения, теоремы, частые ошибки** оформляй admonition-блоками — при
  беглом повторении глаз за них цепляется:
  ```markdown
  !!! note "Определение"
      …
  !!! warning "Частая ошибка"
      …
  ??? example "Пример (раскрывается по клику)"
      …
  ```
- **Перекрёстные ссылки** между билетами — относительными md-ссылками:
  `[см. билет 4](04-slug.md)`. MkDocs сам перепишет их в рабочие веб-ссылки.
  Ссылайся только на существующие файлы (иначе `--strict` упадёт).
- **Картинки** клади в `docs/assets/`, вставляй `![подпись](assets/имя.png)`.
- **Формулы** (если нужны) — LaTeX: `$inline$` и `$$block$$` (MathJax включён).

Создать новый билет из шаблона:
```bash
./scripts/new-bilet.sh 12 graph-coloring "Раскраска графов"
```

---

## Чего НЕ делать

- **Не трогай ветку `gh-pages`** — ею управляет CI, любые ручные правки затрутся.
- **Не коммить папку `site/`** (собранный сайт) — она в `.gitignore`, в репо
  хранится только исходный Markdown.
- **Не пушь, если `mkdocs build --strict` падает.**
- **Не меняй `.github/workflows/` и `mkdocs.yml` по своей инициативе** — только
  по явному запросу. Это инфраструктура, поломка ломает деплой.
- Не добавляй тяжёлых зависимостей в `requirements.txt` без необходимости —
  сборка должна оставаться быстрой.

## Команды-памятка

```bash
pip install -r requirements.txt    # или: uv pip install -r requirements.txt
mkdocs serve                       # предпросмотр с live-reload
mkdocs build --strict              # проверка перед коммитом
./scripts/new-bilet.sh NN slug "Название"   # новый билет
```

---

## Модель деплоя (для понимания, менять не нужно)

```
push main → GitHub Actions (mkdocs build --strict → gh-deploy в ветку gh-pages)
          → GitHub Pages публикует статику на свой CDN (Fastly) с авто-TLS
          → читатель: https://USERNAME.github.io/notes/
```

Сборки на сервере нет — Pages раздаёт готовую статику. Никакого бэкенда,
никакого кластера.
