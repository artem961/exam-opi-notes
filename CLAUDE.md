## Структура билета

Каждый билет — **два раздела**, ничего лишнего:

```markdown
# Билет N. Название вопроса

## Ответ

Ответ как на экзамене — несколько абзацев своими словами, без воды.
Можно вставлять изображения: ![подпись](assets/имя.png)

---

## Подробно

Полное объяснение с нуля для человека, который видит тему впервые.
Примеры, контрпримеры, схемы, изображения — всё уместно.
```

**Правила:**
- Имя файла: `NN-slug.md`, где `NN` — номер с ведущим нулём, `slug` — латиницей через дефис.
- H1 — один, только в самом начале: `# Билет N. Название`.
- Внутри разделов используй H3 (`###`) если нужно дробить — H2 зарезервирован для двух главных разделов.
- Картинки клади в `docs/assets/`, вставляй `![подпись](assets/имя.png)`.
- Формулы — LaTeX: `$inline$` и `$$block$$` (MathJax включён).
- Ссылки на другие билеты — относительными md-ссылками: `[билет 4](04-slug.md)`.
  Ссылайся только на существующие файлы — иначе `--strict` сломает сборку.

## Рабочий цикл

1. Создай/правь файлы в `docs/`.
2. Проверь сборку: `mkdocs build --strict`
3. Коммить и пушь в `main` — CI сам публикует на Pages за ~1–2 мин.

```bash
git add docs/
git commit -m "билет NN: <что сделал>"
git push
```

Создать новый билет из шаблона:
```bash
./scripts/new-bilet.sh 12 graph-coloring "Раскраска графов"
```

## Требования к качеству конспекта

Это экзаменационный конспект, а не учебник и не реферат. Каждое решение по содержанию проверяй вопросом: **«Я написал бы это в ответе на билете?»** Если нет — убирай.

### Что писать

**Раздел «Ответ»** — сжатый, экзаменный ответ своими словами. То, что студент скажет преподавателю за 3–5 минут. Без украшений, без воды.

**Раздел «Подробно»** — объяснение сути с нуля. Целевой читатель: человек, который видит тему впервые. Используй аналогии, простые сравнения, объясняй принцип работы. Каждый новый сложный термин, появляющийся впервые, должен быть объяснён простыми словами сразу при введении.

**Картинки** — обязательны для каждой модели, метода, диаграммы. Если вопрос охватывает несколько объектов (например, три модели ЖЦ), давай отдельную картинку на каждый объект — не одну общую. Картинка должна **визуализировать** теорию, а не дублировать текст, который и так написан рядом. Если нет готового изображения из лекции — нарисуй Mermaid-диаграмму или ASCII-схему прямо в билете.

### Что не писать

**Номера стандартов и ГОСТов** — не пиши коды вида «ISO/IEC 12207:2008» или «ГОСТ Р ИСО/МЭК 12207-2010». Они не запоминаются и не нужны в устном экзаменном ответе. Называй стандарт по смыслу: «международный стандарт жизненного цикла ПО».

**Библиографические ссылки** — не упоминай названия статей, книг, годы публикации, конференции. Запись вида «В статье "Managing the Development of Large Software Systems" (1970)» в конспекте бесполезна.

**Избыточные уточнения** — не пиши пояснения к очевидному. Если написал «группы процессов», не добавляй «(то есть этапы)». Если читатель уже понял слово из контекста — не объясняй его повторно в скобках.

**Воду и историческую лирику** — кто когда что придумал, как развивалась область, общие рассуждения о сложности ПО. Всё это не отвечает на вопрос билета.

### Картинки: приоритеты

1. Слайд из лекции (`docs/assets/lec-*.png`) — использовать, только если он визуализирует именно тот аспект, о котором идёт речь в абзаце рядом.
2. Mermaid-диаграмма — рисуй сам, если нужна схема процесса, последовательности, дерева.
3. ASCII-схема в блоке кода — для простых линейных или иерархических структур.
4. Внешнее изображение — только качественное и наглядное; не брать мутные скриншоты из интернета.
